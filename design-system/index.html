What is a Design System?

A Design System is your team's shared toolkit for creating consistent, high-quality digital products and brand experiences.

Think of it as a well-organised box of Lego bricks, but instead of plastic pieces, you get reusable components, brand assets, design rules, and clear guidelines.

It serves as a single source of truth for both product teams and brand guardians, so no one wastes time reinventing elements or guessing which visual style to use.

Whether you're designing an app screen, a marketing campaign, or a customer touchpoint, the Design System ensures everything looks, feels, and behaves like it comes from the same brand.

What's Inside a Design System?

  • Design tokens — Core values like colour palettes, spacing, typography scales, and animations that power both design and code.
  • Typography — Rules for fonts, sizes, and spacing to ensure readability and brand voice.
  • Components — Reusable interface elements like buttons, forms, and menus.
  • Brand assets — Logos, imagery, and iconography for consistent marketing and product use.
  • Layout principles — Guidelines for arranging content in a clear, intuitive way.

By following a Design System, teams can move faster, collaborate better, and maintain consistency across every brand interaction, whether it's in-app, on the web, or in the real world.

Building a design system from the ground up

A design system is more than a collection of components — it is a shared language between design and engineering. When done well, it reduces inconsistency, speeds up delivery, and creates a foundation that scales with the product. This page demonstrates every typographic element available in the system, rendered with default tokens.

Why tokens matter

Design tokens are the single source of truth for visual decisions. Instead of hardcoding values like font-size: 18px or color: #333, every property references a semantic token. Change the token, and every component updates. The CSS custom property system makes this possible without a build step.

The type scale

Good typography is invisible. Bad typography is everywhere.

The type scale follows a modular progression from --font-3xs through --font-7xl. Body text uses --text-body, which resolves to the primary font at a comfortable reading size. Headings use the secondary font for visual contrast.

  • Headings use the secondary font with tight line height for visual impact
  • Body text uses the primary font with relaxed line height for readability
  • Code and labels use the tertiary font (monospace) for technical precision
  • Spacing between elements follows the spacing scale, not arbitrary values

Keyboard shortcuts

Press Ctrl + Shift + P to open the command palette. Use Esc to dismiss any overlay.

  1. Define brand tokens (fonts, colours, base values)
  2. Build system tokens (scales, semantic mappings)
  3. Create component tokens (button, input, card)
  4. Document everything in the style guide

Small text is used for supporting information, captions, and metadata. It maintains readability at a reduced size with an increased line height to compensate.

Extra-small text is reserved for fine print, legal notices, and tertiary information that should be available but not prominent.

Borders use a composable class system. Start with a structural class (.border, .border-top, etc.), then layer on colour (.border-primary), width (.border-m), and style (.border-dashed) modifiers.

Containers & Max Width

Containers centre content and cap its width. Use .container-small, .container-medium, or .container-large depending on content density. Max-width utilities cap width without centring.

Small container — narrow content like login forms or single-column text
